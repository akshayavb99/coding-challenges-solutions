This commit introduced a breaking interface change to `CutTool`. The constructor
previously accepted `fields` as a `list[int]`. It now expects a `str` (e.g. `"1,2,3"`).

All 18 `test_field_extraction.py` tests instantiate `CutTool` directly with a list:

```python

```

`get_fields()` calls `fields.replace(",", " ")` which raises on a list:

```
AttributeError: 'list' object has no attribute 'replace'
```

## Critical Issues

### 1. `get_fields()` Only Accepts Strings, Not Lists (Line 16)

```python
parts = fields.replace(",", " ").split()
```

The interface change from `list[int]` to `str` was not propagated to the tests.
Either:

**Option A** — update `get_fields()` to handle both types:
```python
if isinstance(fields, list):
    return set(int(f) - 1 for f in fields)
parts = fields.replace(",", " ").split()
```

**Option B** — update all tests to pass strings: `CutTool(fields="2", ...)`.

Option A is safer since it makes `CutTool` more robust to both call sites
(programmatic and CLI).

### 2. `set` Destroys Field Ordering (Line 21)

```python
return set(result)  # remove duplicates
```

Python sets are unordered. `get_result()` then iterates:

```python
selected = [fields[i] for i in self.fields if i < len(fields)]
```

Iteration order over a set is non-deterministic. For `cut -f 3,1`, the output
should always be `city\tname` — but `set` may yield `name\tcity`. Unix `cut`
outputs fields in the order requested on the command line.

**Fix**: Use a dict or a list with manual dedup to preserve order:
```python
seen = set()
ordered = []
for x in result:
    if x not in seen:
        seen.add(x)
        ordered.append(x)
return ordered
```

Or in Python 3.7+, `list(dict.fromkeys(result))` preserves insertion order.

### 3. `test_multiple_fields` Still Fails — Space-Separated Fields Not Supported in CLI

The test invokes:
```
python main.py -f 1 3 file.txt
```

With `-f` now `type=str` (single value), argparse parses `1` as the field string
and `3` and `file.txt` as unrecognized arguments:

```
error: unrecognized arguments: C:\...\test.txt
```

`get_fields()` does handle `"1 2 3"` (space-separated) via the `.replace(",", " ").split()` logic,
but argparse won't consume `3` into the `-f` argument — it stops at a single token.

The test needs to be updated to use comma-separated format: `-f "1,3"`.
