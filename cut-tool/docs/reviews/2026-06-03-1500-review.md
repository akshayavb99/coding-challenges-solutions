## Critical Issues

### 1. `sys` Not Imported (Line 42)

```python
parser.error("Please specify a delimiter or list of fields")
sys.exit(1)   # NameError: name 'sys' is not defined
```

`sys` is never imported. `parser.error()` already calls `sys.exit(2)` internally, so
`sys.exit(1)` is dead code — but it is still a `NameError` waiting to happen if the
`parser.error()` call is ever refactored away. Either add `import sys` or remove line 42.

**Fix**: Remove line 42 entirely. `parser.error()` exits on its own.

### 2. `CutTool.__init__` Crashes When `-d` Given Without `-f` (Line 5)

Validation passes when only `-d` is provided (because `args.delimiter` is truthy). Then:

```python
cut_obj = CutTool(args.fields, args.delimiter, args.file)  # args.fields is None
```

Inside `CutTool.__init__`:
```python
self.fields = [i-1 for i in fields]  # TypeError: 'NoneType' object is not iterable
```

Confirmed by test failure:
```
test_succeeds_with_only_d_flag FAILED
TypeError: 'NoneType' object is not iterable
```

**Fix**: Guard in `__init__`:
```python
self.fields = [i - 1 for i in fields] if fields is not None else []
```

### 3. `argparse` Consumes Positional File Argument as a Field (Lines 34, 36)

`-f` uses `action="extend", nargs="+"` which greedily absorbs all following tokens
until it hits a non-integer — but argparse raises an error instead of stopping cleanly.
When the user runs `python main.py -f 2 file.txt`, argparse tries to parse `file.txt`
as an integer field and errors:

```
error: argument -f/--fields: invalid int value: 'file.txt'
```

Confirmed by three test failures:
```
test_fields_argument_with_file FAILED   (returncode=2)
test_multiple_fields FAILED             (returncode=2)
test_default_delimiter_is_tab FAILED    (returncode=2)
```

This means the tool cannot be invoked in the natural order `main.py -f 2 file.txt`.
The only workaround with the current parser is `main.py file.txt -f 2` (file first)
or `main.py -f 2 -- file.txt` (explicit separator).

**Fix**: Change `-f` to accept a comma-separated string matching Unix `cut` convention:
```python
parser.add_argument('-f', '--fields', action='store', type=str)
```
Then parse the string into a list in `main()` or `CutTool.__init__`:
```python
fields = [int(x) for x in args.fields.split(',')]
```
This also unblocks range support (`1-3`, `2-`) needed in future steps.

## Major Issues

### 4. Validation Condition Not Updated (Line 40)

The prior review requested changing to `args.fields is None and args.delimiter is None`.
The code still uses:
```python
if not(args.fields or args.delimiter):
```

While functionally equivalent for `None` inputs, this is less explicit and will silently
treat `fields=[]` (empty list) as a failure condition if that ever occurs. The user confirmed
the intent is "both cannot be None" — use the explicit form:
```python
if args.fields is None and args.delimiter is None:
```
