### 3. Undefined Variable in Exception Handler (Line 40)
**Severity**: Critical — Runtime error

### 4. No Support for Range Field Specifications
**Severity**: Major — Violates challenge requirements

**File**: cut-tool/cut_tool/main.py (lines 10-23)

**Current implementation**: Only parses comma/space-separated individual fields (e.g., "1,2,3").

**Challenge requirement** (from CLAUDE.md, lines 25-44): Must support range syntax:
- Single fields: `1`
- Ranges: `1-3` (fields 1, 2, 3)
- Open-ended ranges: `2-` (field 2 to end), `-5` (field 1 to 5)
- Mixed: `1,3-5,7` (field 1, then 3-5, then 7)

**Current code treats "1-3" as a single field index and tries to parse it as int**:
```python
f = int(part)  # ValueError: invalid literal for int(): '1-3'
```

**Impact**: Tool cannot handle any range specifications. Step 1-2 may not require ranges, but CLAUDE.md marks this as "High-Risk Area" and challenge scope includes range support.

**Fix**: Add range parsing to `get_fields()`:
```python
def get_fields(self, fields):
    if fields is None:
        return None
    
    result = []
    parts = fields.replace(",", " ").split()
    
    for part in parts:
        if '-' in part:
            # Handle ranges: "1-3", "2-", "-5"
            start, end = part.split('-', 1)
            start = int(start) if start else 1
            end = int(end) if end else None  # Handle "2-" case
            
            if end:
                result.extend(range(start - 1, end))  # Convert to 0-based
            else:
                result.append(start - 1)  # Just the start field
        else:
            result.append(int(part) - 1)
    
    return result
```

### 6. Output Field Order Differs from Real cut
**Severity**: Major — Violates Unix cut semantics

**File**: cut-tool/cut_tool/main.py (lines 34, 35)

**Challenge requirement** (CLAUDE.md): "Evaluate behavior against real cut semantics"

**Problem**: Current implementation preserves output order from `-f` specification:
```python
selected = [fields[i] for i in self.fields if i < len(fields)]
result.append(self.delimiter.join(selected))
```

Example: With `-f 3,1`, output is `city\tname` (spec order), not `name\tcity` (file order).

Real Unix cut always outputs in file order, not specification order. Test at line 182-189 (test_field_order_preserved) INCORRECTLY asserts spec order:
```python
tool = CutTool(fields="3,1,2", ...)
assert result[0] == 'city\tname\tage'  # WRONG - should be 'name\tage\tcity'
```

**Impact**: Tool does not match real cut behavior. Users relying on field reordering will be surprised.

**Fix**: Implement field selection but maintain file order:
```python
selected_indices = set(self.fields)
selected = [f for i, f in enumerate(fields) if i in selected_indices]
```

Also fix the test to expect correct order:
```python
assert result[0] == 'name\tage\tcity'
```

### 8. No Deduplication of Duplicate Field Selections

**Impact**: Tool violates POSIX cut semantics. Users expect deduplication.

**Challenge requirement** (CLAUDE.md, line 85): "duplicate selections" is listed as required test coverage, suggesting expected behavior is unclear. Verify against real cut:

```bash

```

**Fix**: Deduplicate fields while preserving order:
```python
def get_fields(self, fields):
    # ... existing parse logic ...
    seen = set()
    deduped = []
    for f in result:
        if f not in seen:
            deduped.append(f)
            seen.add(f)
    return deduped
```

And fix test:
