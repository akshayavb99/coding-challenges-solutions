## Critical Issue: Duplicates Still Being Removed

The commit attempted to fix set deduplication by replacing `set(result)` with list operations,
but the deduplication logic is still present:

```python
for part in parts:
    f = int(part) - 1
    if f not in result:
        result.append(f)  # <-- Still removes duplicates
```

This check `if f not in result` prevents duplicate field indices from being added.

**Test expectation** (`test_duplicate_field_selection`):
```python
tool = CutTool(fields="1,1,1", ...)
assert result[0] == 'name\tname\tname'  # Expects 3 copies of field 1
```

**Actual behavior**: Only 1 copy is output (duplicates silently removed)

**Unix `cut` behavior**:
```bash
$ echo "a b c" | cut -f 1,1,1 -d ' '
a a a
```

Unix `cut` preserves duplicates. The test is correct; the code is wrong.

**Fix**: Remove the dedup check:
```python
for part in parts:
    f = int(part) - 1
    result.append(f)  # Allow duplicates
```

## Positive Changes

### Set → List Migration
The migration from `set(result)` to list-based storage is correct and addresses the
non-determinism issue. Lists preserve insertion order (guaranteed in Python 3.7+),
while sets do not. This change alone would have fixed the ordering issue.

### Validation Condition
The change from:
```python
if not(args.fields or args.delimiter):
```
to:
```python
if args.fields is None and args.delimiter is None:
```

is more explicit and correct. The original form relied on truthiness, which could
incorrectly reject edge cases like `fields=""` or `fields="0"`. The explicit form
is clearer about the intent: "error only when both are None."
