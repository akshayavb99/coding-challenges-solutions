### Failing Tests (2)

#### 1. `test_duplicate_field_selection` (pre-existing code bug in main.py)

```python
tool = CutTool(fields="1,1,1", delimiter='\t', filename=sample_tab_file)
assert result[0] == 'name\tname\tname'  # FAILS
# Actual: 'name'
```

**Root cause**: `get_fields()` in main.py line 21:
```python
return set(result)  # remove duplicates
```

When parsing `"1,1,1"`, the code builds `[0, 0, 0]` (0-based indices), then converts
to a `set`, yielding `{0}` (single element). Field 1 is output only once.

Unix `cut` preserves duplicates: `cut -f 1,1,1 file` outputs three copies of field 1.

**Fix required in main.py**: Replace `set` with order-preserving deduplication:
```python
seen = set()
ordered = []
for x in result:
    if x not in seen:
        seen.add(x)
        ordered.append(x)
return ordered
```

#### 2. `test_file_permissions_error` (environment limitation)

```python
os.chmod(test_file, 0o000)  # attempt to remove read permission
tool = CutTool(fields="1", ...)
pytest.raises((PermissionError, OSError))  # FAILS
```

**Root cause**: Windows does not honor POSIX `chmod` semantics. `chmod(0o000)` on Windows
does not prevent reads by the file owner. This is a platform limitation, not a code bug.

**Fix**: Skip this test on Windows:
```python
import platform
@pytest.mark.skipif(platform.system() == "Windows", reason="chmod not enforced on Windows")
def test_file_permissions_error(self, tmp_path):
```
